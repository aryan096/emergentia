# Emergentia v0.2 - Development Summary

## Overview

This document summarizes the development progress through v0.2 of Emergentia, a browser-based 2D cellular simulation exploring emergent behavior from local interaction rules.

---

## What Was Built

### Core Infrastructure
- **SvelteKit + Svelte 5** application with runes syntax (`$props`, `$state`, `$effect`)
- **TypeScript** throughout
- **TailwindCSS 4** for styling
- **Canvas 2D rendering** with ImageData for efficient 1-cell-per-pixel rendering

### Simulation Engine

#### Grid System (`Grid.ts`)
- Double-buffered grid with `current` (read) and `next` (write) arrays
- `wasWritten(x, y)` tracking to prevent double-updates in same tick
- `setUnclaimed(x, y, type)` for setting cells without claiming them (allows cascade)
- `setDirect(x, y, type)` for world generation (bypasses double buffering)
- `sync()` to synchronize buffers after world generation

#### Cell Types (`types.ts`)
```typescript
enum CellType {
  Air = 0,
  Water = 1,
  Soil = 2,
  Rock = 3,
  Stone = 4  // Immovable solid for platforms
}
```

#### Density System
```typescript
CELL_DENSITY: { Air: 0, Water: 1, Soil: 2, Rock: 3, Stone: Infinity }
CELL_IS_FLUID: { Air: true, Water: true, Soil: false, Rock: false, Stone: false }
```

#### Simulation Constants
- **Grid size**: 400×250 (100K cells)
- **Tick rate**: 60 ticks/second (fixed)
- **Render FPS**: User-controllable (1-60, default 60)

### Physics Rules

#### Water Rules (`WaterRules.ts`)
Processed bottom→top with randomized horizontal order:
1. **Gravity**: Fall straight down into air
2. **Diagonal flow**: Flow around obstacles diagonally
3. **Pressure spill**: Flow toward columns with lower water height
4. **Extended horizontal flow**: Multi-step splash effect (up to 6 cells)

Key function: `moveWater()` - moves water and leaves source unclaimed as Air

#### Air Rules (`AirRules.ts`)
Processed top→bottom:
1. **Buoyancy**: Air rises through fluids (Water) but NOT through solids (Soil, Rock, Stone)

Key insight: Air only rises through cells where `CELL_IS_FLUID[type] === true`

### World Generation (`WorldGen.ts`)
- Custom **Simplex noise** implementation (`noise.ts`)
- **Fractal Brownian Motion** (fBm) for natural terrain
- Procedural terrain with hills, valleys, caves
- Underground water pools
- Surface lakes in valleys
- Stone containment walls

### Rendering

#### CanvasRenderer (`CanvasRenderer.ts`)
- Direct pixel manipulation via ImageData
- 1 cell = 1 pixel internally
- CSS scaling to 75% viewport width with `image-rendering: pixelated`

#### SimulationCanvas.svelte
- Fixed-interval render loop independent of tick rate
- Reactive to FPS changes

### UI Components

#### ControlPanel.svelte
- Start/Pause, Step, Reset buttons
- Tick counter display
- FPS selector dropdown (1, 2, 5, 10, 15, 30, 60)

## Key Technical Decisions

### Why Double Buffering?
Without it, update order creates artifacts - a cell updated early in a tick sees different neighbors than one updated late.

### Why `wasWritten()` Tracking?
Prevents a cell from being processed multiple times in one tick. When water moves down, the destination is marked as written so it won't be processed again.

### Why `setUnclaimed()`?
When water falls, we want to leave Air behind, but NOT claim that air cell - so water above can cascade into the gap in the same tick.

### Why `getNext()` for availability checks?
**Critical fix!** When checking if a cell is available (isAir), we must read from the NEXT buffer, not CURRENT. Otherwise, vacated cells still appear as Water to cells above, preventing continuous cascades. This was the root cause of excessive air bubbles in water.

### Why Separate Water and Air Rules?
Water must be processed bottom→top (so lower water falls first, creating gaps for upper water). Air must be processed top→bottom (so air bubbles rise properly). Different iteration orders require separate passes.

### Why Fixed Tick Rate + Variable Render FPS?
Simulation physics should run at consistent speed regardless of display. User controls how often they see updates, not how fast physics runs. This also helps with performance - can render at 30 FPS while simulating at 60 ticks/sec.

### Why 400×250 Grid?
Balance between visual interest and performance. Larger grids (800×500, 1000×1000) were too slow. 400×250 = 100K cells at 60 ticks/sec is manageable.

## Known Issues / Ongoing Challenges

### Air Bubbles in Water - SOLVED ✅

Water bodies were showing 50% air mixed in. The root cause was discovered:

**The Bug**: `isAir()` was reading from the CURRENT buffer to check if a cell was available. But when water falls and vacates a cell (via `setUnclaimed`), the CURRENT buffer still shows Water. So water above couldn't see the vacated cell as available.

**The Fix**: `isAir()` now reads from the NEXT buffer via `grid.getNext()`. When a cell is vacated, NEXT immediately shows Air, allowing water above to fall into it in the same tick.

```
Before (broken):                    After (fixed):
Water checks CURRENT[y+1]           Water checks NEXT[y+1]
→ Still shows Water                 → Shows Air (just vacated)
→ Can't fall                        → Falls immediately
→ Air gaps everywhere               → Continuous streams!
```

This is a **critical insight** for any falling-sand simulation with double buffering.

### Performance at Scale
At 300 ticks/sec on 800×500 grid, browser can't keep up. `setInterval` has minimum ~4ms, and each tick processes 400K cells. Solution was reducing grid size and tick rate.

## How to Continue Development

1. Start dev server: `cd emergentia && npm run dev`
2. Grid is 400×250, tick rate is 60/sec
3. World generation is in `generateTestWorld()` - modify for different test scenarios
4. Water rules in `WaterRules.ts`, air rules in `AirRules.ts`
5. Add new cell types: update `CellType` enum, `CELL_REGISTRY`, `CELL_DENSITY`, `CELL_IS_FLUID`
6. Add new rules: create new rule file, call from `Simulation.tick()`

## Session Context for Future LLM Agents

This project uses:
- **Svelte 5 runes syntax** - use `$state`, `$props`, `$effect`, NOT old `let` reactivity
- **TailwindCSS 4** - may have different syntax than v3
- **Custom noise** - no external noise libraries, see `utils/noise.ts`
- **Bottom-up water iteration** - critical for proper water physics
- **Top-down air iteration** - critical for proper bubble rise

The core challenge is balancing:
- Physical realism (water should pool, not have air gaps)
- Emergence (no scripted behavior, only local rules)
- Performance (100K cells × 60 ticks/sec = 6M cell updates/sec)
